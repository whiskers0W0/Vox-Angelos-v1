# Urgency Algorithm — Location Density Score (PostGIS)

## What was clarified

The panel asked for the *technical execution* of the Location Density Score to be
specified — concretely, whether distance/clustering calculations use a geospatial
database extension (PostGIS) so the system doesn't crash or degrade once there are
thousands of pins. Prior to this change, `Concern` only stored plain
`double? Latitude` / `double? Longitude` columns with no distance calculation
anywhere in the codebase (the Heatmap page just passed raw lat/lng to the client-side
map for visual clustering — nothing was computed server-side). This document describes
what was actually built to answer that gap.

## Technical execution

**Database**: Postgres (via `Npgsql.EntityFrameworkCore.PostgreSQL`, confirmed in
`Program.cs`), which makes PostGIS a native fit — no separate spatial database needed.

**What was added**:

1. **`Npgsql.EntityFrameworkCore.PostgreSQL.NetTopologySuite`** package, enabling
   `NetTopologySuite.Geometries.Point` as a first-class EF Core column type mapped to
   Postgres's `geography` type. Enabled via `options.UseNpgsql(connectionString, o =>
   o.UseNetTopologySuite())` in `Program.cs`.
2. **`Concern.Location`** (`VoxAngelos/Data/Concerns.cs`) — a
   `geography (Point, 4326)` column (SRID 4326 = standard lat/lng, WGS84), kept in sync
   with the existing `Latitude`/`Longitude` columns rather than replacing them (so
   nothing that reads `Latitude`/`Longitude` today — the Heatmap page's map rendering —
   needed to change).
3. **A GIST index on `Location`** (`ApplicationDbContext.OnModelCreating`,
   `.HasIndex(c => c.Location).HasMethod("GIST")`) — this is the piece that actually
   prevents the crash/slowdown the panel was concerned about. Without a spatial index,
   "how many concerns are near this pin" is an O(n) scan computing distance to every
   other row; with a GIST index over a `geography` column, Postgres can prune the
   search to a bounding region first, so the query stays fast as the table grows into
   the thousands of rows.
4. **`Concern.LocationDensityScore`** (`int`) — the actual score, described below.
5. **The `AddLocationDensityScore` migration** (`VoxAngelos/Migrations/`) additionally
   runs `CREATE EXTENSION IF NOT EXISTS postgis;` (auto-generated by EF Core once
   `UseNetTopologySuite()` is enabled) and backfills `Location` + an initial
   `LocationDensityScore` for any concerns that existed before this change.

## The formula

`LocationDensityScore` = count of **other** concerns that are:
- within **300 meters** of this concern's pin (`DensityRadiusMeters` in
  `UrgencyScoreService`),
- submitted within the last **30 days** (`DensityWindowDays`),
- not yet `Resolved`.

Computed with PostGIS's `ST_DWithin(geography, geography, meters)` — this is the
PostGIS-recommended way to do a radius search on `geography` columns, because it can
use the GIST index directly (unlike computing `ST_Distance` for every row and
filtering afterward, which forces a full scan).

```sql
SELECT COUNT(*) FROM "Concerns" n
WHERE n."Id" != c."Id"
  AND n."Location" IS NOT NULL
  AND n."SubmittedAt" >= now() - interval '30 days'
  AND n."Status" != 'Resolved'
  AND ST_DWithin(c."Location", n."Location", 300)
```

This is intentionally the *density* half only — a count of corroborating nearby
reports, which is the strongest, least-assumption-laden urgency signal available (many
independent citizens reporting problems in the same small area over a short window is
a meaningful real-world urgency indicator regardless of category). Category-severity
weighting or a single blended "UrgencyScore" number was deliberately **not** invented
on top of this, since no such formula previously existed in the codebase or thesis
manuscript to match — `LocationDensityScore` is exposed as its own explicit,
independently-inspectable number so it can be combined with whatever additional
weighting the manuscript ultimately specifies, without this implementation guessing
wrong on the panel's behalf.

## When it's computed

`UrgencyScoreService.ApplyLocationAsync(concern)` runs once, right after a new concern
is saved (`Pages/User/Create.cshtml.cs`, `OnPostAsync`):

1. Sets the new concern's `Location` from its `Latitude`/`Longitude`.
2. Computes the new concern's own `LocationDensityScore` (one `ST_DWithin` count).
3. **Recomputes every existing neighbor's `LocationDensityScore` too** — a new report
   arriving near an older one should bump the older one's density score immediately,
   not just its own. This is also a single `ST_DWithin`-filtered `UPDATE`, so it stays
   index-backed even though it touches multiple rows.

All three steps are plain `UPDATE`/`SELECT` statements executed via
`ExecuteSqlInterpolatedAsync` — no rows are pulled into .NET memory and looped over, so
the cost of "how dense is this area" scales with what the GIST index can prune to, not
with the total number of concerns in the table.

## Where it's visible

- `LGU/Index.cshtml` — a "🔥 N nearby reports" badge on each concern card when its
  density score is above zero (visible to LGU staff triaging their queue).
- `LGU/Heatmap.cshtml.cs` — `locationDensityScore` is included in the JSON payload sent
  to the map, available for the front-end to weight/color pins by density in addition
  to raw position.

## Demonstrating it for defense

1. Submit 3–4 concerns with pins within ~300m of each other (the seeded test accounts'
   barangays, or any nearby coordinates on the map picker) within a short time window.
2. Show each concern's density badge incrementing on the LGU dashboard as each new one
   lands — including the *earlier* concerns' badges updating, proving the neighbor
   recompute step runs, not just the newly-submitted one.
3. Optionally, `EXPLAIN ANALYZE` the `ST_DWithin` query directly against the
   `Concerns` table to show the query planner using the `IX_Concerns_Location` GIST
   index rather than a sequential scan.
